# Sync filters with the map on every page that has both

## Scope (pages with both a filter control AND a WFS map)
- **Explore/Index** — chip filter + map. Already synced; refactor to a shared helper.
- **Trip/Index** (builder) — planning filter bar + `#tripBuilderMap`. Markers NOT synced → fix.
- **NearMe/Index** — type/rating/distance/search chips + `#nearbyMap`. Map shows ALL branches, list is server-filtered → fix.
- **Trip/Details** — has a map but NO filter control → out of scope (per constraint).
- All other `filter-bar` pages (Destination, Tourist, Sponsor, Reward, Mission, etc.) have **no WFS map** → out of scope.
- `Destination`/`SponsorBranch` Create/Edit use `initLocationPicker` (single-point picker, no list filter) → out of scope.

## Reusable helper (do this first)
Extend `wwwroot/js/maps.js` `initWfsMap` so the geojson layer + features are captured once and exposed via a generic `filterMarkers(predicateFn)`.

In `initWfsMap`, after `geojsonLayer` is built:
- Keep an internal array `var markerEntries = [];` and push `{ layer: l, feature: feature }` in `onEachFeature` (Leaflet sets `l.feature = feature`, so `feature` is available).
- Return object gains `filterMarkers`:
  ```js
  filterMarkers: function (predicate) {
      if (!markerEntries.length) return;
      markerEntries.forEach(function (entry) {
          var show = (typeof predicate !== 'function') ? true : predicate(entry.feature, entry.layer);
          if (show) { if (!map.hasLayer(entry.layer)) entry.layer.addTo(map); }
          else { if (map.hasLayer(entry.layer)) map.removeLayer(entry.layer); }
      });
  }
  ```
- The existing `layer()` accessor can stay. `initLocationPicker` is untouched.

This reuses Explore's exact show/hide mechanism (`map.addLayer`/`removeLayer`) and is the single source of truth.

## 2. Trip/Index.cshtml — wire Filter → map
Add `data-dest-id="@Model.Stops[i].DestinationId"` to each `.picker-row` (id is already in the hidden `Stops[i].DestinationId` input; expose it as a data-attr for matching).
- Build `var rowById = {}` from `.picker-row` (key = `data-dest-id`).
- Extract a reusable `function rowMatches(row)` containing the SAME match logic currently inline in `applyTripFilter` (interests via `data-category`/`data-status`; budget via `data-price` <= budget). Keep `applyTripFilter` calling `rowMatches` for the list (behavior unchanged).

- **Duration / Companions**: intentionally NOT part of `rowMatches`/`tripPredicate` — they remain trip metadata only, consistent with the existing list behavior.

## 3. NearMe/Index.cshtml — sync branch markers to the server-filtered sponsor list
NearMe filtering is **server-side** (postback via `applyNearMeFilter`); keep that list behavior unchanged. The map must reflect the same filtered set. The list shows sponsors; the map shows branches; each card carries `data-id` = `Sponsor.Id`, and branch GeoJSON features carry `sponsor_id`/`SponsorId`.
- In `initWfsMap` `onLayerReady`, read the visible sponsor ids and filter:
  ```js
  var visibleSponsorIds = Array.prototype.slice
      .call(document.querySelectorAll('#nearMeList .nearme-card[data-id]'))
      .map(function (el) { return el.getAttribute('data-id'); });
  nearMeMap.filterMarkers(function (f) {
      var sid = _firstDefined(f.properties, ['sponsor_id', 'SponsorId', 'sponsorId']);
      return visibleSponsorIds.indexOf(String(sid)) > -1;
  });
  ```
- Because each filter is a postback, the page reloads, the list re-renders filtered, the WFS re-fetches all branches, and `onLayerReady` re-syncs the map. This satisfies "map matches the list" without changing list behavior. (Note: the "immediate hide/show without re-fetch" guidance applies to Explore/Trip where data is client-side; NearMe's postback model means the map re-syncs after reload — acceptable and within constraints.)
- Edge: a sponsor passing the distance filter may still show its other (farther) branches; acceptable for visual sync. WFS `maxFeatures` cap (50) may omit some branches — pre-existing limitation, not changed.
