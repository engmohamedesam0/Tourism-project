# UI Layout Fixes — Admin nav clutter, stat-box legibility, top spacing

Source of truth for implementation. All changes are CSS/HTML (Razor) only — no controller,
route, or data changes. Verify by running the app and checking the three roles called out
in the deliverable.

## Root-cause findings (from code inspection)

### 1. Admin nav
- `Views/Shared/_Layout.cshtml` Admin block (~lines 372–437) **leaks tourist pages**:
  "Near Me" (~416–420) and "Features" (~421–425) sit *inside* the
  `@if (User.IsInRole("Admin"))` block. Tourist/Guest blocks correctly contain them
  (~477–486, ~517–525) — so they are duplicated into Admin by mistake.
- Admin currently lists ~8–10 standalone top-level items (Tourists, Destinations, Trip Plans,
  Missions, Rewards dropdown, Near Me, Features, Accounts, Support Inbox, + the
  `AdminApprovalBell` component). Too crowded.

### 2. Stat-box illegibility + height
- `_Layout.cshtml` defines `.stat-card` **twice**:
  - First block (~40–97): compact **white** card — `padding: 22px 22px 20px`,
    `.stat-label { color:#6b7280 }` (clearly legible muted gray), 46px colored-square icon.
  - Second block (~109–175): **dark hero** variant — `padding: 45px 25px 35px`,
    `background: rgba(255,255,255,0.04)`, `.stat-label { color: rgba(255,255,255,0.55) }`
    (light translucent → invisible on the light `#FDF6EC` page bg unless text-selected).
- The dark block is NOT scoped, so it **overrides** the white block for *every* `.stat-card`,
  including inner-page data cards. `stats-section` (the only place the dark variant belongs)
  is used **only** in `Home/Index.cshtml` (~43, 268, 276, 280). All other `_StatBoxRow`
  usages (Tourist, Destination, Mission, Reward, Sponsor*, TripPlan, AdminSupport, etc.) are
  inner pages → they get the dark variant wrongly.

### 3. Top-of-page gap
- Two-tier fixed navbar: upper `.navbar-utility` (~31px, driven by the sm search input) +
  lower `.navbar-primary` (~36px, brand + nav-link padding). Actual rendered height ≈ **70–72px**.
- `.inner-page-wrapper` has a stale `padding-top: 140px` (both in `_Layout.cshtml` ~181 and
  `site.css`). 140px >> ~71px → large empty gap.
- **Double-offset bug**: `_Layout` already wraps every non-home page in
  `<main class="inner-page-wrapper">` (~560–567), BUT many views ALSO wrap their own content
  in `<div class="inner-page-wrapper">` (About, Login, Register, SponsorBranch/*,
  SponsorReward/*, SponsorReview/Index — 13 views). Nested `.inner-page-wrapper` →
  `padding-top` applied twice → ~280px gap on those pages.
- Custom full-page layouts add their OWN hard-coded `140px` offset and
  `calc(100vh - 140px)` heights: `Explore/Index.cshtml` (139, 145, 261) and
  `NearMe/Index.cshtml` (155, 161, 242) → `140 (layout) + 140 (own) = 280px`.
- Trip/Index `min-height` (320/340px) are content min-heights, not top gaps — leave them.

## Implementation

### Fix 1 — Admin nav (`Views/Shared/_Layout.cshtml`)
1. Inside the Admin block, **delete** the "Near Me" and "Features" `<li>`s (~416–425).
2. Replace the inline Tourists/Destinations/Trip Plans/Missions items with a **Manage**
   dropdown (same dropdown-item pattern already used by Rewards).
3. Keep the existing **Rewards** dropdown as-is.
4. Add an **Administration** dropdown: Accounts, Support Inbox, Approvals. Put the
   pending-approvals badge on the dropdown **toggle** (Fix 1c).
5. Remove the standalone `@await Component.InvokeAsync("AdminApprovalBell")` `<li>` (~436)
   and invoke the component inside the Administration toggle instead.

Resulting Admin top-level items: **Home** (standalone) · **Manage ▾** · **Rewards ▾** ·
**Administration ▾** → 4 top-level entries (within the ~5 budget).

#### 1c. Badge on the Administration toggle (`Views/Shared/Components/AdminApprovalBell/Default.cshtml`)
- Change the component view to render **only the badge** (no `<li>`), so it embeds in the toggle:
  ```razor
  @model Tourist_Project_MVC.View_Model.AdminApprovalBellVM
  @if (Model.PendingCount > 0)
  {
      <span class="badge rounded-pill text-bg-danger ms-1">@Model.PendingCount</span>
  }
  ```
- Administration toggle in `_Layout`:
  ```razor
  <a class="nav-link dropdown-toggle" href="#" id="adminDropdown" role="button"
     data-bs-toggle="dropdown" aria-expanded="false">
      <i class="bi bi-gear-fill"></i> Administration
      @await Component.InvokeAsync("AdminApprovalBell")
  </a>
  <ul class="dropdown-menu dropdown-menu-dark dropdown-menu-end" aria-labelledby="adminDropdown">
      <li><a class="dropdown-item" asp-controller="Role" asp-action="ManageAccounts"><i class="bi bi-person-gear me-2"></i> Accounts</a></li>
      <li><a class="dropdown-item" asp-controller="AdminSupport" asp-action="Index"><i class="bi bi-headset me-2"></i> Support Inbox</a></li>
      <li><a class="dropdown-item" asp-controller="SponsorApproval" asp-action="Index"><i class="bi bi-bell-fill me-2"></i> Approvals</a></li>
  </ul>
  ```
  (Component is used only in `_Layout`, so changing its view is safe.)

### Fix 2 — Stat-box legibility + height (`Views/Shared/_Layout.cshtml`)
Scope the **dark hero** `.stat-card` block (~109–175) under `.stats-section` so it only applies
on the home hero, leaving inner pages with the compact white style. Prefix every selector in
that block with `.stats-section `:
- `.stat-card` → `.stats-section .stat-card`
- `.stat-card::before` → `.stats-section .stat-card::before`
- `.stat-card:hover` → `.stats-section .stat-card:hover`
- `.stat-card:hover::before` → `.stats-section .stat-card:hover::before`
- `.stat-icon` → `.stats-section .stat-icon`
- `.stat-number` → `.stats-section .stat-number`
- `.stat-suffix` → `.stats-section .stat-suffix`
- `.stat-label` → `.stats-section .stat-label`

Effect: inner-page `.stat-card` uses the first block (white bg, `padding:22px`,
`.stat-label{color:#6b7280}`, 46px colored-square icon) → legible labels + compact height,
applied everywhere `_StatBoxRow` is used (one shared fix). Home hero keeps the dark variant.

### Fix 3 — Top spacing (layout-level, site-wide)
Introduce a `--nav-offset` CSS variable = actual navbar height.

**`wwwroot/css/site.css`**
- In `:root` add `--nav-offset: 80px;` (safe static fallback ≥ real height so content is
  never hidden if JS is unavailable).
- `.inner-page-wrapper { padding-top: 140px; ... }` → `padding-top: var(--nav-offset); ...`.
- Add: `.inner-page-wrapper .inner-page-wrapper { padding-top: 0 !important; }`
  (neutralizes the 13 self-wrapping views site-wide without editing them).
- Reduce `.inner-page-wrapper .container { padding-top: 30px; ... }` → `padding-top: 16px;`.

**`Views/Shared/_Layout.cshtml`**
- Inline `.inner-page-wrapper { padding-top: 140px; ... }` (~181) → `padding-top: var(--nav-offset); ...`.
- Add the same `.inner-page-wrapper .inner-page-wrapper { padding-top: 0 !important; }` rule.

**`wwwroot/js/scripts.js`**
- Add `syncNavOffset()` that sets
  `document.documentElement.style.setProperty('--nav-offset', mainNav.offsetHeight + 'px')`
  (measured at top of page where both bars are visible → exact full height). Call it on
  `DOMContentLoaded`, `window.load`, and `resize` so the offset precisely matches the two-tier
  navbar in both scroll states (offset region scrolls away when scrolled, so no gap/jump).

**`Views/Explore/Index.cshtml`** (139, 145, 261)
- `.explore-page { padding-top: 140px; }` → `padding-top: 0;` (removes the duplicate 140px).
- `calc(100vh - 140px)` (×2) → `calc(100vh - var(--nav-offset))`.

**`Views/NearMe/Index.cshtml`** (155, 161, 242)
- `.nearme-page { padding-top: 140px; }` → `padding-top: 0;`
- `calc(100vh - 140px)` (×2) → `calc(100vh - var(--nav-offset))`.

(Register/About/Login/SponsorBranch*/SponsorReward*/SponsorReview are self-wrapping and are
fixed by the `.inner-page-wrapper .inner-page-wrapper { padding-top:0 }` rule — no per-page edit.)
