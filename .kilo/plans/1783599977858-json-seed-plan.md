Replace ad-hoc migration `HasData` with a maintainable, idempotent JSON-based seeding routine that populates every table with varied Egypt-themed sample data.

## JSON schema rules
- **Explicit IDs across all JSON files** for readability and cross-file FK clarity.
- **Nullable FK fields** are omitted or `null` when not applicable.
- **Dates** use ISO-8601 strings (`"2026-08-01"` or `"2026-08-01T00:00:00"`).
- **Decimals** use JSON numbers (`25.00`).
- **users.json** is special: it describes ApplicationUser profiles, intended role, and seed password. No `Id` collisions with Identity.

## Identity seeding (users.json → UserManager / RoleManager)
- Read `users.json` entries.
- Ensure roles exist: `Admin`, `User`, `Sponsor` via `RoleManager.RoleExistsAsync` / `CreateAsync`.
- For each user entry:
  - `await _userManager.FindByEmailAsync(email)` — skip if exists.
  - `await _userManager.CreateAsync(user, seedPassword)` with explicit `Id` set.
  - `await _userManager.AddToRoleAsync(user, role)`.
- Tourist and Sponsor tables get their rows linked via `ApplicationUserId` in the same routine.

## Non-Identity seeding strategy
All app tables use SQL Server **identity** integer PKs (confirmed in `20260709083200_mig.cs`). To preserve explicit IDs from JSON while keeping inserts safe:

### Approach: IDENTITY_INSERT per table
For each table T in FK-dependency order:
```csharp
if (!context.Set<T>().Any())
{
    context.Database.ExecuteSqlRaw("SET IDENTITY_INSERT [T] ON");
    context.Set<T>().AddRange(entities);
    context.SaveChanges();
    context.Database.ExecuteSqlRaw("SET IDENTITY_INSERT [T] OFF");
}
```
- This satisfies the explicit-ID requirement and works with existing identity columns.
- Idempotency is guaranteed by the `Any()` guard.
- No transaction-wrap needed (each table is independent; partial seeds self-heal on next startup).

## Open questions / risks
- **Existing migration seed**: Tables already populated by `TouristContext.OnModelCreating` (Destinations 1-15, Sponsors 1-5, etc.) will be skipped by `Any()` checks. If the goal is to OVERWRITE old seed with richer JSON, the migration `HasData` blocks that on existing DBs. **Recommendation**: keep migration `HasData` as a baseline; new JSON seed only fills empty tables. To refresh data, clear tables first.
- **Identity password policy**: Current config requires 12-char passwords. Seed passwords must comply.
- **DateTime kind**: JSON dates should be unambiguous (use `DateTimeKind.Unspecified` or UTC consistently).
